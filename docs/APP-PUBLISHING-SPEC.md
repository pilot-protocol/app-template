# Pilot App Publishing — Standard Flow (SPEC)

> Status: **CANONICAL (RC1).** Reconciles the original design (`docs/AGENT-APPS.md`)
> with what shipped, and pins the standardized publishing flow. Decisions for
> RC1: **self-signed publishers, gated by catalogue PR + CI** (no Pilot
> countersign yet); **a `<ns>.help` discovery method is required** of every app.
> Items marked *PLANNED* are specified here but land in a later RC.

This is the single source of truth for how an app goes from a developer's
existing API/CLI to `pilotctl appstore install`. If behavior and this doc
disagree, that's a bug in one of them — file it.

---

## 1. Definitions

| Term | Meaning |
|---|---|
| **App** | A capability installable by id (`io.pilot.<name>`), reverse-DNS. |
| **Adapter** | The thin, stateless binary that speaks app-store IPC and forwards to a backend. Generated by `pilot-app`. |
| **Backend** | The real service the adapter forwards to — an HTTP API (today) or a local CLI (PLANNED, needs `proc.exec`). Holds all state. |
| **Manifest** | `manifest.json` — the signed declaration of id, version, exposed methods, grants, and publisher. The *only* source of grant truth. |
| **Bundle** | The release tarball: `manifest.json` (sha-pinned + signed) + `bin/<binary>`. |
| **Catalogue** | `catalogue/catalogue.json` — the index the daemon reads to discover installable apps. |
| **Publisher** | The holder of an ed25519 key that signs a manifest. Self-asserted in RC1. |

---

## 2. Where things live (topology)

```
pilot-protocol/app-template     PUBLIC — the tool, this SPEC, submission CI, publish automation, the single front door
pilot-protocol/catalog          PUBLIC — bundle tarballs as GitHub Release assets ("CDN") + publishers/registry.json
TeoSlayer/pilotprotocol@main    catalogue/catalogue.json  (the index — data entries only)
~/.pilot/apps/<id>/             install root on each host (manifest.json, bin/, app.sock, supervisor.log)
```

**Invariants:**
- App bundles release on `pilot-protocol/catalog`, **never** on
  `TeoSlayer/pilotprotocol` — releases there drive the platform installer's
  "latest" and would shadow the platform binaries.
- The platform repo's *only* app-store footprint is `catalogue.json` **data
  entries** (one line per app), written by automation. No app-store tooling, CI,
  or templates live in the platform repo — all of that lives in `app-template`.
  The gate runs in `app-template` (§7) before any catalogue entry is opened, so
  a second in-platform-repo gate would be redundant.

## 3. The canonical flow

```
 AUTHOR ─────────────▶ BUILD ─────────▶ SUBMIT ─────▶ REVIEW (you) ─▶ PUBLISH ─▶ INSTALL ─▶ DISCOVER ─▶ CALL
 pilot.app.yaml        pilot-app init   pilot-app      catalogue PR    gh release  pilotctl    <ns>.help    pilotctl
                       make package     submit         + CI gate       + JSON PR   install                  call
```

### 3.1 Author
Write `pilot.app.yaml` (§5.1). One file: id, version, backend URL, methods →
endpoints, grants. `pilot-app validate` gives fast feedback.

### 3.2 Build
`pilot-app init` scaffolds the adapter; `make package`:
1. `go build` the adapter,
2. sha256-**pin** the binary into a copy of the manifest,
3. ed25519-**sign** the manifest (signing covers `publisher:id:manifest_version:binary.sha256:grants-hash`),
4. `tar` the bundle.

Signing happens **after** pinning — the binary hash is in the signed payload.

### 3.3 Submit (single front door)
`pilot-app submit -C . --prepare <app-template-fork>` runs the **same checks CI
will run** locally, then writes a submission payload into a fork of the single
central repo `pilot-protocol/app-template`: `submissions/<id>/` containing the
signed bundle + a `submission.json`. The publisher commits that and opens **one
PR** — to `app-template`. They never need push to any org repo, and never touch
the platform repo or the catalogue directly.

### 3.4 Review (the gate)
The **submission PR on `app-template`** is the gate. Two layers, both required:
- **CI** (§7) — `submission-validate` runs `pilot-app verify` on the bundle:
  manifest parses + `Validate()`s + signature verifies, binary sha pinned,
  `exposes` contains a `<ns>.help` method, no unknown caps, id/version consistent.
- **Human** (§7.2) — maintainer checklist: publisher identity, grant
  proportionality (especially any `proc.exec`/`fs.write`/`key.sign`), description
  accuracy, backend ownership.

RC1 trust model: publishers are **self-signed**; the submission PR review + CI is
the trust boundary. `TrustedPublishers` enforcement (cryptographic allowlist) is
a documented future toggle (§8), not required to ship.

### 3.5 Publish (automated)
On merge to `app-template`, the `publish-on-merge` automation re-verifies the
bundle, releases it on `pilot-protocol/catalog`, and opens the one-line
`catalogue.json` PR on `TeoSlayer/pilotprotocol`. Once a maintainer merges that
data PR, hosts pick the app up on next `catalogue`/`install`. The platform repo's
only change is that one catalogue data entry.

### 3.6 Install (agent host)
```
pilotctl appstore catalogue           # discover (reads catalogue.json; override $PILOT_APPSTORE_CATALOG_URL)
pilotctl appstore install <id>        # HTTPS GET bundle_url → verify bundle_sha256 → extract to ~/.pilot/apps/<id>/
                                       # → supervisor rescans (≤30s) → re-verify manifest sig + binary sha → spawn → ready
```
Integrity is re-checked on **every** spawn: tarball sha at install, binary
sha + signature on each launch. A mismatch refuses to spawn and audits both
hashes.

### 3.7 Discover (agent)
The agent learns the method surface by calling the **required** discovery method:
```
pilotctl appstore call <id> <ns>.help '{}'
```
which returns every method with params, `kind`, duration class, and measured
roundtrip (§5.4). `appstore list`/`status` show the flat `exposes` names.
*PLANNED:* a rendered `SKILL.md` push surface (§8) layered on the same metadata.

### 3.8 Call
```
pilotctl appstore call <id> <method> '<json>'   # brokered over the app's unix socket; JSON in → JSON out
```

### 3.9 Update
Bump `app_version` for every new binary (the supervisor keys respawn on version;
re-releasing the same version won't roll hosts). New release + catalogue PR (new
`bundle_sha256`). The supervisor refuses downgrades. Changing **grants** bumps
`manifest_version` → re-consent at upgrade (§5.2, G9).

### 3.10 Revoke *(PLANNED, G5)*
A `revoked.json` deny-list on `pilot-protocol/catalog`, fetched by the daemon and
honored by `install`/spawn. Until shipped, revocation = remove the catalogue
entry + delete the release (stops new installs; doesn't reach installed hosts).

## 4. Trust & integrity (RC1)

Two cryptographic facts, both verified at install and every spawn:
- **`bundle_sha256`** in `catalogue.json` pins the tarball — a swapped CDN byte fails.
- **`binary.sha256` + ed25519 signature** in the manifest — `VerifySignature`
  (`pkg/manifest`) confirms the manifest was signed by the key it names, and the
  supervisor recomputes the binary hash on each launch.

What RC1 does **not** do: confirm the publisher key is *trusted* (the
`VerifyTrustAnchor` allowlist is empty / not enforced in the supervisor scan).
That gap is closed socially by the PR review, not cryptographically. §8 tracks
turning on the allowlist when desired.

## 5. Artifacts & schemas

### 5.1 `pilot.app.yaml`
See `pilot-app example`. Required: `id`, `app_version`, `description`,
`backend.{type,base_url|command}`, `methods[]`. Each method maps to an HTTP
route (`http: {verb, path}`) or a CLI invocation (`cli: {args}`), with `kind`,
`duration`, optional `timeout`/`params`/`roundtrip`.

### 5.2 `manifest.json`
Schema is `app-store/pkg/manifest`. Generated, never hand-edited. Key rules:
- `id` reverse-DNS; `app_version` semver; `manifest_version ≥ 1`.
- `binary.runtime ∈ {go,bun,node,python}`; `binary.sha256` 64-hex (pinned at build).
- `exposes` lists every IPC method **and must include `<ns>.help`** (§5.4).
- `grants` ⊆ known caps (§5.3); at least one grant.
- `manifest_version` increments **only** when grants change → triggers re-consent.

### 5.3 Capabilities (grants)
Known today: `fs.read fs.write fs.append fs.delete net.dial net.call ipc.call
key.sign audit.log`. Standard adapter grant set:
- HTTP: `net.dial → <backend-host>` (rate-limited) + `fs.read $APP/config.json` + `audit.log`.
- CLI *(PLANNED)*: `proc.exec → <executable>` — requires adding the cap (§8, G6).

### 5.4 Discovery contract — `<ns>.help` (REQUIRED)
Every app **must** expose `<namespace>.help`, a local (no-backend) method returning:
```json
{ "app": "io.pilot.x", "version": "0.1.0", "description": "...",
  "duration_classes": {"fast":"...","med":"...","slow":"..."},
  "methods": [ {"method":"x.foo","kind":"utility","summary":"...",
                "params":{"q":"string (required)"},
                "duration":"fast","typical_roundtrip":"~0.5s"} ] }
```
The generator emits this automatically. CI rejects any app whose `exposes`
lacks a `.help` method (§7).

### 5.5 `catalogue.json` entry
```json
{ "id":"io.pilot.x", "version":"0.1.0",
  "description":"one line, accurate",
  "bundle_url":"https://github.com/pilot-protocol/catalog/releases/download/<ns>-v0.1.0/io.pilot.x-0.1.0.tar.gz",
  "bundle_sha256":"<64-hex of the tarball>" }
```
Top level: `{"version":1,"updated_at":"<RFC3339>","apps":[ … ]}`. Preserve
existing entries; one entry per app (latest version).

### 5.6 Publisher registry *(RC1 identity, G4)*
`pilot-protocol/catalog/publishers/registry.json`: maps publisher pubkey → human
identity + contact, so a reviewer can tie a signature to a known party.
Identity only in RC1 — not yet a cryptographic gate.
```json
{ "publishers": [ {"pubkey":"ed25519:…","name":"Alex Godoroja","contact":"alex@vulturelabs.io","apps":["io.pilot.cosift"]} ] }
```

## 6. `pilot-app submit`

Single-front-door submission helper.
1. Re-runs §7.1 checks on the built bundle (pre-flight).
2. `--prepare <fork>` writes `submissions/<id>/` (bundle + `submission.json`) into
   a fork/checkout of `pilot-protocol/app-template`; the publisher commits + PRs it.
3. Without `--prepare`, it prints the underlying release + catalogue steps for the
   **org-maintainer direct path** (no client needs this).

---

## 7. The review gate

### 7.1 CI checks (objective — `submission-validate` workflow in `app-template`)
Runs `pilot-app verify` on each submitted bundle (and the publish automation
re-runs it before opening any catalogue PR). Checks:
1. `bundle_url` resolves (HTTP 200) and downloads.
2. `sha256(tarball) == bundle_sha256`.
3. Tarball contains `manifest.json` + `bin/<binary>`.
4. `manifest.binary.sha256 == sha256(bin/<binary>)`.
5. `manifest.Validate()` passes (id/semver/caps/grants well-formed).
6. `manifest.VerifySignature()` passes (signed by the key it names).
7. `exposes` contains exactly one `*.help` method (discovery contract).
8. `catalogue.id == manifest.id` and `catalogue.version == manifest.app_version`.
9. Version is **not a downgrade** of the entry it replaces.

A live check (install + call `<ns>.help`) runs in the author's local loop and is
recommended but not required in CI (needs a daemon).

### 7.2 Human reviewer checklist
- [ ] Publisher in `publishers/registry.json` (or added in the same PR).
- [ ] Grants are proportional to the described function. **Scrutinize**
      `proc.exec`, `fs.write`, `key.sign`, broad `net.dial` targets.
- [ ] `description` matches what the app actually does.
- [ ] Backend host is one the publisher controls.
- [ ] No release on `TeoSlayer/pilotprotocol` (bundle is on `catalog`).

## 8. Gap status (every gap, with disposition)

| Gap | RC1 disposition | Owner / where |
|---|---|---|
| **G1** Submission package | **DONE** — `pilot-app submit` | `cmd/pilot-app` |
| **G2** Review gate | **DONE** — submission PR template + checklist (§7.2) | `app-template/.github` |
| **G3** Catalogue CI | **DONE** — `submission-validate` workflow (verifier) | `app-template/.github/workflows` |
| **G4** Publisher identity | **DONE (identity)** — `publishers/registry.json` (§5.6) | `pilot-protocol/catalog` |
| **G4′** Trust anchor (crypto) | *DEFERRED* — populate `TrustedPublishers` + enforce `VerifyTrustAnchor` in supervisor scan when desired | `app-store/pkg/manifest`, `plugin/appstore/supervisor.go` |
| **G5** Revocation | *PLANNED* — `revoked.json` deny-list (§3.10) | `app-store` + `catalog` |
| **G6** `proc.exec` (CLI apps) | *PLANNED* — add cap to `KnownCaps`; declaration-consent now, sandbox later | `app-store/pkg/manifest/validate.go`; see `docs/CLI-ADAPTER.md` |
| **G7** Discovery contract | **DONE** — `<ns>.help` required + CI-checked (§5.4) | generator + CI |
| **G8** Sandbox enforcement | *DEFERRED (RC2)* — seccomp/namespaces | `app-store` (tracked in its CHANGELOG) |
| **G9** Upgrade grant re-consent | *VERIFY/PLANNED* — surface grant-diff on `manifest_version` bump at install | `pilotctl` install path |

"DONE" = shipped in this tooling layer (the scaffolder + catalogue gate).
"PLANNED/DEFERRED" = specified here with the exact change point; not required for
RC1 publishing to be standardized and safe under the PR+CI trust model.

## 9. Quick reference

```bash
go install github.com/pilot-protocol/app-template/cmd/pilot-app@latest
pilot-app example > pilot.app.yaml && $EDITOR pilot.app.yaml
pilot-app validate
pilot-app init -o ./my-app
cd my-app && make gen-key && make package
pilot-app submit -C . --prepare /path/to/app-template-fork   # writes submissions/<id>/
# commit + PR to pilot-protocol/app-template; CI + maintainer gate; merge → auto-publish
pilotctl appstore install io.pilot.my-app
pilotctl appstore call io.pilot.my-app my-app.help '{}'
```
